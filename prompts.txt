Update SparkVM to use a Debian minbase rootfs as the default managed VM image.

Goal:
Remove static language runtimes and Docker-export based rootfs creation.

SparkVM should use one lightweight Debian minbase rootfs:

~/.sparkvm/images/debian-rootfs.ext4

Users install dependencies through rollout setup_cmd and execute through run_cmd.

Network support is NOT required in this phase.
We will implement networking later.

Important:
Do not use Docker to build rootfs.
Use debootstrap.

Firecracker still uses:
- managed Firecracker binary
- managed kernel image
- managed Debian rootfs image
- rollout execution disk

Default managed layout:

~/.sparkvm/
├── bin/
│   └── firecracker
├── images/
│   ├── vmlinux
│   └── debian-rootfs.ext4
├── rollouts/
│   ├── metadata.json
│   └── rollout-*/
├── workers/
└── cache/

Public user flow:

sparkvm setup

Then:

from sparkvm.rollouts import Rollouts
from sparkvm import SparkVM

rollouts = Rollouts()

rollout = rollouts.create(
    name="repo-test",
    mode="repo",
    source="./my-repo",
    setup_cmd="apt-get update && apt-get install -y python3 python3-pip && pip3 install -r requirements.txt",
    run_cmd="pytest -q",
    disk_mb=8192,
)

result = SparkVM(vcpu=2, memory="4G", timeout=900).run(rollout.id)

print(result.stdout)
print(result.stderr)
print(result.exit_code)

Important limitation:
Since VM networking is not implemented yet, setup_cmd commands that require internet access may fail inside the VM.
Document this clearly.
For now, users can:
- vendor dependencies inside the repo
- use local files in rollout
- wait for networking support

Do not fake networking support.

5. Debian rootfs builder
--------------------------------------------------

Implement rootfs build helper in setup.py or image builder module.

Function:

build_debian_minbase_rootfs(
    *,
    output_path: Path,
    size_mb: int = 2048,
    suite: str = "bookworm",
    mirror: str = "http://deb.debian.org/debian",
    force: bool = False,
) -> Path

Behavior:

1. If output_path exists and force=False:
   - reuse it
   - return output_path

2. Create temp build directory under:
   ~/.sparkvm/cache/build-debian-*/

3. Create rootfs directory:
   <temp>/rootfs

4. Run debootstrap:

   sudo debootstrap \
     --variant=minbase \
     --include=ca-certificates,curl,iproute2,procps,coreutils,util-linux \
     bookworm \
     <temp>/rootfs \
     http://deb.debian.org/debian

Notes:
- ca-certificates/curl are useful later when network is implemented.
- iproute2 is needed later for guest network config.
- util-linux/coreutils provide common tools.
- Do not install python/node/go by default.
- Keep the rootfs minimal.

5. Inject /init:
   write <temp>/rootfs/init
   chmod +x <temp>/rootfs/init

6. Ensure basic files exist:
   - /bin/sh
   - /init
   - /sbin/poweroff or equivalent
   - /bin/mount or /usr/bin/mount

7. Create ext4 image:

   dd if=/dev/zero of=<output_path>.tmp bs=1M count=<size_mb>
   mkfs.ext4 -F <output_path>.tmp

8. Mount image:

   sudo mount <output_path>.tmp <mount_dir>

9. Copy rootfs into mounted image:

   sudo cp -a <temp>/rootfs/. <mount_dir>/

10. Sync and unmount:

   sync
   sudo umount <mount_dir>

11. Atomic rename:

   mv <output_path>.tmp <output_path>

12. Cleanup temp dirs even on failure where safe.

Errors:
- Missing debootstrap -> SparkVMSetupError
- debootstrap failure -> SparkVMSetupError
- mount failure -> SparkVMSetupError
- copy failure -> SparkVMSetupError
- unmount failure -> CleanupError or SparkVMSetupError

Important:
Do not leave mounted filesystems behind.
Use finally blocks for unmount cleanup.

6. SparkVM /init for Debian rootfs
--------------------------------------------------

Inject this /init into the Debian rootfs:

#!/bin/sh
set +e

mount -t proc proc /proc
mount -t sysfs sysfs /sys
mount -t devtmpfs devtmpfs /dev || true

mkdir -p /job
mount /dev/vdb /job

mkdir -p /job/results

cd /job

if [ -f /job/setup.sh ]; then
  sh /job/setup.sh > /job/results/setup.stdout.log 2> /job/results/setup.stderr.log
  setup_code=$?
else
  setup_code=0
fi

echo "$setup_code" > /job/results/setup.exit_code

if [ "$setup_code" -ne 0 ]; then
  echo "$setup_code" > /job/results/final_exit_code
  sync
  poweroff -f
fi

sh /job/run.sh > /job/results/run.stdout.log 2> /job/results/run.stderr.log
run_code=$?

echo "$run_code" > /job/results/run.exit_code
echo "$run_code" > /job/results/final_exit_code

sync
poweroff -f

Important:
- /init must not assume Python exists.
- /init must not assume network exists.
- /init must only run setup.sh if it exists.
- /init must always run run.sh after successful setup.
- /init must write phase logs into /job/results.
- /init must power off the VM.

    name: str
    stdout: str
    stderr: str

    rollout_id: str

    rollout_mode: str

- stderr returns run.stderr if run exists else ""

- setup_failed if setup.exit_code != 0
- run_failed if run.exit_code != 0
- passed if final_exit_code == 0

- timeout for timeout
- oom for likely OOM

Goal:
Remove Debian minbase rootfs build from `sparkvm setup`.

1. `sparkvm setup` only initializes SparkVM directories, Firecracker binary, and kernel image.

4. Dockify command
--------------------------------------------------

Add CLI:

sparkvm dockify <docker-image>

Options:
- --name <runtime-name>
- --size-mb <int>, default 2048
- --force
- --pull / --no-pull
- --home-dir <path>

Examples:
sparkvm dockify python:3.12-slim
sparkvm dockify node:22-slim
sparkvm dockify ubuntu:24.04 --name ubuntu-light
sparkvm dockify ghcr.io/org/custom:latest --size-mb 4096

Behavior:

1. Verify Docker CLI exists.
2. If --pull is enabled, run:
   docker pull <docker-image>
3. Create container:
   docker create <docker-image>
4. Export filesystem:
   docker export <container> into temp rootfs directory.
5. Remove container.
6. Inject SparkVM /init at:
   <rootfs>/init
7. chmod +x <rootfs>/init
8. Validate rootfs basics:
   - /bin/sh exists
   - /init exists and executable
   - mount command exists if possible
   - shutdown/poweroff/halt/busybox exists if possible
9. Create ext4 image:
   ~/.sparkvm/images/<runtime-name>.ext4
10. Mount ext4 image.
11. Copy rootfs into mounted ext4 image.
12. Unmount.
13. Write metadata:
   ~/.sparkvm/images/<runtime-name>.json

Metadata example:

{
  "runtime": "python-3.12-slim",
  "source_image": "python:3.12-slim",
  "rootfs": "/home/user/.sparkvm/images/python-3.12-slim.ext4",
  "size_mb": 2048,
  "created_at": "2026-05-18T12:00:00Z",
  "init_injected": true
}

Cleanup:
- Always remove Docker container.
- Always unmount image on failure.
- Remove temporary image on failure.
- Do not leave mounted filesystems.

Privileges:
Dockify may need root for mount/umount.
Do not call interactive sudo.
If current process is not root and mount requires root, fail clearly:

"Dockify requires root privileges for mounting ext4 images.
Run:
  sudo <absolute sparkvm executable> dockify <image> --home-dir /home/<user>/.sparkvm --owner <user>"

Support:
--owner USER
If running with sudo and owner is provided:
- chown the generated .ext4 and .json to owner
- chown created files under ~/.sparkvm/images as needed

#!/bin/sh
set +e

shutdown_vm() {
  sync
  if command -v poweroff >/dev/null 2>&1; then
    poweroff -f
  fi
  if command -v halt >/dev/null 2>&1; then
    halt -f
  fi
  if command -v reboot >/dev/null 2>&1; then
    reboot -f
  fi
  echo "SparkVM: no shutdown command found" > /dev/console
  while true; do sleep 3600; done
}

mount -t sysfs sysfs /sys
mount -t devtmpfs devtmpfs /dev || true

mkdir -p /job
mount /dev/vdb /job

mkdir -p /job/results

cd /job

if [ -f /job/setup.sh ]; then
  sh /job/setup.sh > /job/results/setup.stdout.log 2> /job/results/setup.stderr.log
  setup_code=$?
else
  setup_code=0
fi

echo "$setup_code" > /job/results/setup.exit_code

if [ "$setup_code" -ne 0 ]; then
  echo "$setup_code" > /job/results/final_exit_code
  shutdown_vm
fi

sh /job/run.sh > /job/results/run.stdout.log 2> /job/results/run.stderr.log
run_code=$?

echo "$run_code" > /job/results/run.exit_code
echo "$run_code" > /job/results/final_exit_code

shutdown_vm

4. Network support design
--------------------------------------------------

Implement TAP/NAT based networking.

Add new file:

sparkvm/network.py

Create dataclass:

@dataclass(frozen=True)
class NetworkConfig:
    enabled: bool
    tap_name: str
    guest_mac: str
    host_ip: str
    guest_ip: str
    guest_cidr: str
    subnet_cidr: str
    out_iface: str
    dns: str = "1.1.1.1"

Create class:

class NetworkManager:
    def __init__(self, *, home_dir: Path) -> None:
        ...

    def setup(self, vm_id: str) -> NetworkConfig:
        ...

    def cleanup(self, config: NetworkConfig) -> None:
        ...

Network topology:
- One TAP device per VM.
- Host side IP: 172.30.X.1/30
- Guest side IP: 172.30.X.2/30
- Subnet: 172.30.X.0/30
- X allocated from vm_id hash or random free range.
- TAP name must fit Linux IFNAMSIZ limit, max 15 chars.
  Example: spk<8-char-id>
  For vm_id vm-02e67edfc7a0:
    tap name could be spk02e67edf

Host setup steps:
1. Detect default outbound interface:
   ip route get 1.1.1.1
   parse "dev <iface>"

2. Create TAP:
   ip tuntap add dev <tap_name> mode tap

3. Assign host IP:
   ip addr add <host_ip>/30 dev <tap_name>

4. Bring TAP up:
   ip link set dev <tap_name> up

5. Enable IPv4 forwarding:
   sysctl -w net.ipv4.ip_forward=1

6. Add NAT:
   iptables -t nat -A POSTROUTING -s <subnet_cidr> -o <out_iface> -j MASQUERADE

7. Add forwarding:
   iptables -A FORWARD -i <tap_name> -o <out_iface> -j ACCEPT
   iptables -A FORWARD -i <out_iface> -o <tap_name> -m conntrack --ctstate RELATED,ESTABLISHED -j ACCEPT

8. Block cloud metadata endpoint:
   iptables -A FORWARD -i <tap_name> -d 169.254.169.254 -j REJECT

Cleanup steps:
Reverse exactly:
1. Delete NAT rule with -D.
2. Delete FORWARD rules with -D.
3. Delete metadata block rule with -D.
4. Delete TAP device:
   ip link delete <tap_name>

Important:
- Do not leave stale TAP devices.
- Cleanup should be best-effort but must raise CleanupError if it cannot clean and no prior exception exists.
- If cleanup fails after an original run error, preserve original error and include cleanup failure in failure.json if possible.

Privileges:
Network setup needs root/CAP_NET_ADMIN.

Behavior:
- If network=True and current user cannot create TAP or iptables rules, raise NetworkSetupError with clear message.
- Do not prompt for sudo password.
- Do not call interactive sudo.
- For now, simplest accepted behavior:
  network=True requires the SparkVM process to run with root privileges or the required capabilities.
- Error message:
  "Network setup requires root/CAP_NET_ADMIN for TAP and iptables. Run with sudo or configure capabilities."

Later we can add a privileged helper service, but not in this phase.
