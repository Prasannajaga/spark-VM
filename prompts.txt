Update SparkVM to use a Debian minbase rootfs as the default managed VM image.

Goal:
Remove static language runtimes and Docker-export based rootfs creation.

SparkVM should use one lightweight Debian minbase rootfs:

~/.sparkvm/images/debian-rootfs.ext4

Users install dependencies through rollout setup_cmd and execute through run_cmd.

Network support is NOT required in this phase.
We will implement networking later.

Important:
Do not use Docker to build rootfs.
Use debootstrap.

Firecracker still uses:
- managed Firecracker binary
- managed kernel image
- managed Debian rootfs image
- rollout execution disk

Default managed layout:

~/.sparkvm/
├── bin/
│   └── firecracker
├── images/
│   ├── vmlinux
│   └── debian-rootfs.ext4
├── rollouts/
│   ├── metadata.json
│   └── rollout-*/
├── workers/
└── cache/

Public user flow:

sparkvm setup

Then:

from sparkvm.rollouts import Rollouts
from sparkvm import SparkVM

rollouts = Rollouts()

rollout = rollouts.create(
    name="repo-test",
    mode="repo",
    source="./my-repo",
    setup_cmd="apt-get update && apt-get install -y python3 python3-pip && pip3 install -r requirements.txt",
    run_cmd="pytest -q",
    disk_mb=8192,
)

result = SparkVM(vcpu=2, memory="4G", timeout=900).run(rollout.id)

print(result.stdout)
print(result.stderr)
print(result.exit_code)

Important limitation:
Since VM networking is not implemented yet, setup_cmd commands that require internet access may fail inside the VM.
Document this clearly.
For now, users can:
- vendor dependencies inside the repo
- use local files in rollout
- wait for networking support

Do not fake networking support.

5. Debian rootfs builder
--------------------------------------------------

Implement rootfs build helper in setup.py or image builder module.

Function:

build_debian_minbase_rootfs(
    *,
    output_path: Path,
    size_mb: int = 2048,
    suite: str = "bookworm",
    mirror: str = "http://deb.debian.org/debian",
    force: bool = False,
) -> Path

Behavior:

1. If output_path exists and force=False:
   - reuse it
   - return output_path

2. Create temp build directory under:
   ~/.sparkvm/cache/build-debian-*/

3. Create rootfs directory:
   <temp>/rootfs

4. Run debootstrap:

   sudo debootstrap \
     --variant=minbase \
     --include=ca-certificates,curl,iproute2,procps,coreutils,util-linux \
     bookworm \
     <temp>/rootfs \
     http://deb.debian.org/debian

Notes:
- ca-certificates/curl are useful later when network is implemented.
- iproute2 is needed later for guest network config.
- util-linux/coreutils provide common tools.
- Do not install python/node/go by default.
- Keep the rootfs minimal.

5. Inject /init:
   write <temp>/rootfs/init
   chmod +x <temp>/rootfs/init

6. Ensure basic files exist:
   - /bin/sh
   - /init
   - /sbin/poweroff or equivalent
   - /bin/mount or /usr/bin/mount

7. Create ext4 image:

   dd if=/dev/zero of=<output_path>.tmp bs=1M count=<size_mb>
   mkfs.ext4 -F <output_path>.tmp

8. Mount image:

   sudo mount <output_path>.tmp <mount_dir>

9. Copy rootfs into mounted image:

   sudo cp -a <temp>/rootfs/. <mount_dir>/

10. Sync and unmount:

   sync
   sudo umount <mount_dir>

11. Atomic rename:

   mv <output_path>.tmp <output_path>

12. Cleanup temp dirs even on failure where safe.

Errors:
- Missing debootstrap -> SparkVMSetupError
- debootstrap failure -> SparkVMSetupError
- mount failure -> SparkVMSetupError
- copy failure -> SparkVMSetupError
- unmount failure -> CleanupError or SparkVMSetupError

Important:
Do not leave mounted filesystems behind.
Use finally blocks for unmount cleanup.

6. SparkVM /init for Debian rootfs
--------------------------------------------------

Inject this /init into the Debian rootfs:

#!/bin/sh
set +e

mount -t proc proc /proc
mount -t sysfs sysfs /sys
mount -t devtmpfs devtmpfs /dev || true

mkdir -p /job
mount /dev/vdb /job

mkdir -p /job/results

cd /job

if [ -f /job/setup.sh ]; then
  sh /job/setup.sh > /job/results/setup.stdout.log 2> /job/results/setup.stderr.log
  setup_code=$?
else
  setup_code=0
fi

echo "$setup_code" > /job/results/setup.exit_code

if [ "$setup_code" -ne 0 ]; then
  echo "$setup_code" > /job/results/final_exit_code
  sync
  poweroff -f
fi

sh /job/run.sh > /job/results/run.stdout.log 2> /job/results/run.stderr.log
run_code=$?

echo "$run_code" > /job/results/run.exit_code
echo "$run_code" > /job/results/final_exit_code

sync
poweroff -f

Important:
- /init must not assume Python exists.
- /init must not assume network exists.
- /init must only run setup.sh if it exists.
- /init must always run run.sh after successful setup.
- /init must write phase logs into /job/results.
- /init must power off the VM.

--------------------------------------------------

Keep phase-based result collection.

Read from execution disk:

/results/setup.stdout.log
/results/setup.stderr.log

VMResult:

@dataclass(frozen=True)
class PhaseResult:
    name: str
    stdout: str
    stderr: str
    exit_code: int

@dataclass(frozen=True)
class VMResult:
    rollout_id: str
    rollout_name: str
    rollout_mode: str
    base_image: str
    vm_id: str
    status: str
    exit_code: int
    duration_ms: int

    timed_out: bool = False
    oom_killed: bool = False
    firecracker_log_path: Path | None = None
    execution_disk_path: Path | None = None

Properties:
- stdout returns run.stdout if run exists else ""
- stderr returns run.stderr if run exists else ""
- passed returns exit_code == 0 and not timed_out and not oom_killed

Status rules:
- setup_failed if setup.exit_code != 0
- run_failed if run.exit_code != 0
- passed if final_exit_code == 0
- infrastructure_failed for boot/disk/api failures
- timeout for timeout
- oom for likely OOM

Update SparkVM runtime image strategy.

Goal:
Remove Debian minbase rootfs build from `sparkvm setup`.

New model:
1. `sparkvm setup` only initializes SparkVM directories, Firecracker binary, and kernel image.

4. Dockify command
--------------------------------------------------

Add CLI:

sparkvm dockify <docker-image>

Options:
- --name <runtime-name>
- --size-mb <int>, default 2048
- --force
- --pull / --no-pull
- --home-dir <path>

Examples:
sparkvm dockify python:3.12-slim
sparkvm dockify node:22-slim
sparkvm dockify ubuntu:24.04 --name ubuntu-light
sparkvm dockify ghcr.io/org/custom:latest --size-mb 4096

Behavior:

1. Verify Docker CLI exists.
2. If --pull is enabled, run:
   docker pull <docker-image>
3. Create container:
   docker create <docker-image>
4. Export filesystem:
   docker export <container> into temp rootfs directory.
5. Remove container.
6. Inject SparkVM /init at:
   <rootfs>/init
7. chmod +x <rootfs>/init
8. Validate rootfs basics:
   - /bin/sh exists
   - /init exists and executable
   - mount command exists if possible
   - shutdown/poweroff/halt/busybox exists if possible
9. Create ext4 image:
   ~/.sparkvm/images/<runtime-name>.ext4
10. Mount ext4 image.
11. Copy rootfs into mounted ext4 image.
12. Unmount.
13. Write metadata:
   ~/.sparkvm/images/<runtime-name>.json

Metadata example:

{
  "runtime": "python-3.12-slim",
  "source_image": "python:3.12-slim",
  "rootfs": "/home/user/.sparkvm/images/python-3.12-slim.ext4",
  "size_mb": 2048,
  "created_at": "2026-05-18T12:00:00Z",
  "init_injected": true
}

Cleanup:
- Always remove Docker container.
- Always unmount image on failure.
- Remove temporary image on failure.
- Do not leave mounted filesystems.

Privileges:
Dockify may need root for mount/umount.
Do not call interactive sudo.
If current process is not root and mount requires root, fail clearly:

"Dockify requires root privileges for mounting ext4 images.
Run:
  sudo <absolute sparkvm executable> dockify <image> --home-dir /home/<user>/.sparkvm --owner <user>"

Support:
--owner USER
If running with sudo and owner is provided:
- chown the generated .ext4 and .json to owner
- chown created files under ~/.sparkvm/images as needed

4. Network support design
--------------------------------------------------

Implement TAP/NAT based networking.

Add new file:

sparkvm/network.py

Create dataclass:

@dataclass(frozen=True)
class NetworkConfig:
    enabled: bool
    tap_name: str
    guest_mac: str
    host_ip: str
    guest_ip: str
    guest_cidr: str
    subnet_cidr: str
    out_iface: str
    dns: str = "1.1.1.1"

Create class:

class NetworkManager:
    def __init__(self, *, home_dir: Path) -> None:
        ...

    def setup(self, vm_id: str) -> NetworkConfig:
        ...

    def cleanup(self, config: NetworkConfig) -> None:
        ...

Network topology:
- One TAP device per VM.
- Host side IP: 172.30.X.1/30
- Guest side IP: 172.30.X.2/30
- Subnet: 172.30.X.0/30
- X allocated from vm_id hash or random free range.
- TAP name must fit Linux IFNAMSIZ limit, max 15 chars.
  Example: spk<8-char-id>
  For vm_id vm-02e67edfc7a0:
    tap name could be spk02e67edf

Host setup steps:
1. Detect default outbound interface:
   ip route get 1.1.1.1
   parse "dev <iface>"

2. Create TAP:
   ip tuntap add dev <tap_name> mode tap

3. Assign host IP:
   ip addr add <host_ip>/30 dev <tap_name>

4. Bring TAP up:
   ip link set dev <tap_name> up

5. Enable IPv4 forwarding:
   sysctl -w net.ipv4.ip_forward=1

6. Add NAT:
   iptables -t nat -A POSTROUTING -s <subnet_cidr> -o <out_iface> -j MASQUERADE

7. Add forwarding:
   iptables -A FORWARD -i <tap_name> -o <out_iface> -j ACCEPT
   iptables -A FORWARD -i <out_iface> -o <tap_name> -m conntrack --ctstate RELATED,ESTABLISHED -j ACCEPT

8. Block cloud metadata endpoint:
   iptables -A FORWARD -i <tap_name> -d 169.254.169.254 -j REJECT

Cleanup steps:
Reverse exactly:
1. Delete NAT rule with -D.
2. Delete FORWARD rules with -D.
3. Delete metadata block rule with -D.
4. Delete TAP device:
   ip link delete <tap_name>

Important:
- Do not leave stale TAP devices.
- Cleanup should be best-effort but must raise CleanupError if it cannot clean and no prior exception exists.
- If cleanup fails after an original run error, preserve original error and include cleanup failure in failure.json if possible.

Privileges:
Network setup needs root/CAP_NET_ADMIN.

Behavior:
- If network=True and current user cannot create TAP or iptables rules, raise NetworkSetupError with clear message.
- Do not prompt for sudo password.
- Do not call interactive sudo.
- For now, simplest accepted behavior:
  network=True requires the SparkVM process to run with root privileges or the required capabilities.
- Error message:
  "Network setup requires root/CAP_NET_ADMIN for TAP and iptables. Run with sudo or configure capabilities."

Later we can add a privileged helper service, but not in this phase.

Update SparkVM worker/runtime/result handling with four critical fixes:

1. Treat runtime rootfs images as immutable.
2. Prevent env secret leakage while still printing phase logs into firecracker.log.
3. Preserve workers for every failed/non-passed execution.
4. Never copy results back into persistent rollouts.

Context:
SparkVM currently uses dockified runtime rootfs images under:

~/.sparkvm/images/<runtime>.ext4

Rollouts live under:

~/.sparkvm/rollouts/<rollout-id>/

Workers live under:

~/.sparkvm/workers/<vm-id>/

Execution currently uses:
- runtime rootfs
- rollout execution disk
- /init inside guest
- setup.sh and run.sh
- result files under /job/results

The implementation must clearly separate:

Persistent input:
  ~/.sparkvm/rollouts/<rollout-id>/

Persistent runtime image:
  ~/.sparkvm/images/<runtime>.ext4

Per-run mutable state:
  ~/.sparkvm/workers/<vm-id>/

--------------------------------------------------
1. Runtime rootfs immutability
--------------------------------------------------

Problem:
The dockified runtime image should not be mutated by a VM run.
If Firecracker attaches ~/.sparkvm/images/<runtime>.ext4 as writable, then:
- the base image can be corrupted
- concurrent runs can conflict
- permission problems happen
- setup_cmd can permanently modify the runtime image

Required design:
Runtime image under ~/.sparkvm/images must be treated as immutable.

Implement per-run rootfs copy:

For each SparkVM.run():

1. Resolve base runtime image:

   ~/.sparkvm/images/<runtime>.ext4

2. Create worker dir:

   ~/.sparkvm/workers/<vm-id>/

3. Create per-run rootfs copy:

   ~/.sparkvm/workers/<vm-id>/rootfs.ext4

4. Copy base runtime rootfs into worker rootfs.

Preferred copy command:

   cp --reflink=auto --sparse=always <base-rootfs> <worker-rootfs>

Fallback:
- if reflink is not supported, normal copy is acceptable.

5. Attach worker rootfs as the VM root device.

Important:
- Do NOT attach ~/.sparkvm/images/<runtime>.ext4 directly as writable.
- The base runtime image must remain unchanged after every run.

Firecracker drive behavior:
- rootfs drive path should be:
  ~/.sparkvm/workers/<vm-id>/rootfs.ext4
- rootfs drive may be attached read-write because it is a per-run copy.
- rollout execution disk remains:
  ~/.sparkvm/workers/<vm-id>/rollout.ext4
- rollout execution disk is attached read-write.

Why not attach base rootfs read-only only?
Because SparkVM supports setup_cmd, and setup_cmd may install dependencies into the rootfs.
A strictly read-only rootfs would break apt/pip/npm installs.
So the correct tradeoff is:
- base runtime image = immutable
- per-worker rootfs copy = writable
- worker is deleted on success, preserved on failure

Add helper in disk or image module:

def create_worker_rootfs(
    *,
    base_rootfs: Path,
    worker_rootfs: Path,
) -> Path:
    ...

Behavior:
- check base_rootfs exists
- check base_rootfs readable
- copy using cp --reflink=auto --sparse=always
- chmod worker_rootfs 0600 or 0644
- ensure current user can read/write worker_rootfs
- return worker_rootfs

Add preflight:
- base rootfs only needs readable permission
- worker rootfs must be readable and writable
- rollout.ext4 must be readable and writable

Add errors:
- RuntimeImagePermissionError
- WorkerRootfsError

Acceptance:
- Runtime image file is never modified during run.
- Firecracker uses workers/<vm-id>/rootfs.ext4, not images/<runtime>.ext4.
- setup_cmd can modify the per-run rootfs.
- success deletes worker rootfs.
- failure preserves worker rootfs for debugging.

Add function:

prepare_linux_runtime() {
  mkdir -p /proc /sys /dev /dev/pts

  mountpoint -q /proc || mount -t proc proc /proc
  mountpoint -q /sys || mount -t sysfs sysfs /sys
  mountpoint -q /dev || mount -t devtmpfs devtmpfs /dev

  mkdir -p /dev/pts
  mountpoint -q /dev/pts || mount -t devpts devpts /dev/pts

  ln -sf /proc/self/fd /dev/fd
  ln -sf /proc/self/fd/0 /dev/stdin
  ln -sf /proc/self/fd/1 /dev/stdout
  ln -sf /proc/self/fd/2 /dev/stderr
  ln -sf /proc/kcore /dev/core 2>/dev/null || true

  mkdir -p /tmp /run /var/tmp
  mountpoint -q /tmp || mount -t tmpfs tmpfs /tmp || true
  mountpoint -q /run || mount -t tmpfs tmpfs /run || true
  mountpoint -q /var/tmp || mount -t tmpfs tmpfs /var/tmp || true

  export PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin
  export DEBIAN_FRONTEND=noninteractive
  export TZ=Etc/UTC
}

Reason:
Many runtimes require a normal Linux process environment:
- /proc for process/interpreter discovery
- /sys for device/kernel info
- /dev for devices
- /dev/pts for apt/dpkg/terminal-backed tools
- /dev/fd and stdio links for language runtimes and shells
- /tmp, /run, /var/tmp for temporary writes
- stable PATH for scripts

This is not Python-specific.

configure_network() {
  if [ ! -f /job/.sparkvm/network.env ]; then
    return 0
  fi

  . /job/.sparkvm/network.env

  if [ "${SPARKVM_NET_ENABLED:-0}" != "1" ]; then
    return 0
  fi

  if ! command -v ip >/dev/null 2>&1; then
    echo "SparkVM: ip command missing; network unavailable" > /dev/console
    return 0
  fi

  ip link set eth0 up
  ip addr add "$SPARKVM_GUEST_CIDR" dev eth0
  ip route add default via "$SPARKVM_HOST_IP" dev eth0

  if [ -n "${SPARKVM_DNS:-}" ]; then
    echo "nameserver ${SPARKVM_DNS}" > /etc/resolv.conf 2>/dev/null || true
  else
    echo "nameserver 1.1.1.1" > /etc/resolv.conf 2>/dev/null || true
  fi
}

This is generic.
It does not assume any language runtime.

redact_to_console() {
  file="$1"

  if [ ! -s "$file" ]; then
    return 0
  fi

print_phase_logs() {

  out_file="/job/results/${phase}.stdout.log"

run_phase() {
  phase="$1"
  script="$2"

  echo "SparkVM: running ${script}" > /dev/console

  sh "$script" > "/job/results/${phase}.stdout.log" 2> "/job/results/${phase}.stderr.log"
  code=$?

  echo "SparkVM: ${script} exit code=${code}" > /dev/console
  echo "$code" > "/job/results/${phase}.exit_code"

  print_phase_logs "$phase"

  return "$code"
}

  setup_code=$?

if [ "$setup_code" -ne 0 ]; then

8. Shutdown function
--------------------------------------------------

Keep robust shutdown_vm:

shutdown_vm() {
  sync

  if command -v poweroff >/dev/null 2>&1; then
    poweroff -f >/dev/null 2>&1 &
  fi
  if command -v halt >/dev/null 2>&1; then
    halt -f >/dev/null 2>&1 &
  fi
  if command -v reboot >/dev/null 2>&1; then
    reboot -f >/dev/null 2>&1 &
  fi
  if command -v busybox >/dev/null 2>&1; then
    busybox poweroff -f >/dev/null 2>&1 &
    busybox reboot -f >/dev/null 2>&1 &
  fi

  sleep 1

  if [ -w /proc/sysrq-trigger ]; then
    echo s > /proc/sysrq-trigger || true
    echo u > /proc/sysrq-trigger || true
    echo o > /proc/sysrq-trigger || true
    sleep 1
    echo b > /proc/sysrq-trigger || true
  fi

  echo "SparkVM: no shutdown command succeeded" > /dev/console
  while true; do sleep 3600; done
}
